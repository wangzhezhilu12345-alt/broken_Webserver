# Broken Server

一个用 C++ 从零练手实现的简易 HTTP 服务器项目，代码按 `v1`、`v2`、`v3` 三个阶段逐步演进。项目的核心目标，是把“能访问网页”这件事一步步拆开，从最基础的 HTTP 请求解析，到多连接处理，再到基于 Reactor + `epoll` 的事件驱动模型，慢慢搭出一个更像样的服务端框架。

这个仓库不是现成框架封装出来的 Demo，而是带着明显学习轨迹的版本迭代项目。很多设计思路、踩坑记录和版本总结，都来自作者自己的实践过程。目前最完整的版本是 `v3`。

## 项目目标

结合 [v3/broken_src/项目需求分析&&框架设计.md](/home/ubuntu/broken_server/v3/broken_src/项目需求分析&&框架设计.md) 里的思路，这个项目主要围绕两层展开：

- 业务层：处理 HTTP 请求、设计请求解析类、设计响应构造类。
- 底层：采用 Reactor 思想，分别处理 IO、计时器和信号。

在 `v3` 中，项目进一步把服务拆成了更清晰的职责分层：

- HTTP 层：解析请求、构造响应。
- IO 层：负责连接、读写、事件分发。
- 逻辑层：拿到请求后查找页面资源并返回结果。
- 网络存储层：目前还是预留概念，后续可以扩展数据库或其他后端服务。

整体调用链路可以概括为：

`HTTP层 -> IO层 -> 逻辑层 -> IO层 -> HTTP层`

## 版本演进

### `v1`

最初版本，完成了最基本的 HTTP 访问能力。

- 能解析基础请求并返回页面
- 整体还是单进程、单线条处理思路
- 更偏向“先跑起来”的原型验证

### `v2`

开始处理多连接，并引入线程池思想。

- 支持多个客户端连接
- 服务器开始区分 IO、信号、计时器这些职责
- 每个连接对应一个 `fd`，再包装成对象作为上下文传递
- 线程池负责并发处理任务，主线程负责投递工作

这个阶段的重点，是把“单进程能跑”升级成“多连接也能处理”。

### `v3`

当前主版本，核心思路是 Reactor + `epoll` + 逻辑层解耦。

- 主线程持续运行事件循环
- IO 层和逻辑层通过请求队列、结果队列交付任务
- 线程池不再和主线程并行抢职责，而是专注处理业务逻辑
- 页面解析与静态文件返回职责更清晰
- 已补充测试首页和常见错误页：`404`、`403`、`500`、`504`
- 支持可选守护进程启动：`./server --daemon`

如果你只想体验当前版本，直接看 `v3` 即可。

## 目录结构

```text
broken_server/
├── epollpra/            # epoll 相关练习代码
├── v1/                  # 第一个可访问版本
├── v2/                  # 支持多连接和线程池的版本
└── v3/                  # 当前主版本
    ├── broken_include/  # 公共头文件
    ├── http/            # HTTP 请求解析与响应构造
    ├── io/              # socket、connection、epoll 等 IO 组件
    ├── logic/           # 逻辑层、线程池、页面分发
    ├── server/          # 服务主循环
    └── broken_src/      # 启动入口、静态页面、makefile
```

## `v3` 当前能力

- 基础 HTTP 请求处理
- 静态页面返回
- 基于 `epoll` 的事件驱动
- 线程池处理逻辑任务
- 信号与计时器事件接入事件循环
- 自定义错误页返回
- 可选守护进程启动

现在 `v3/broken_src` 下已经提供了一个简单测试首页，方便直接验证：

- `/`：测试首页
- `/missing-page`：触发真实 `404`
- `/404.html`：404 页面预览
- `/504.html`：504 页面预览

## 编译与运行

建议在 `v3/broken_src` 目录下操作，因为当前静态资源路径依赖工作目录。

### 编译

```bash
cd v3/broken_src
make
```

### 前台运行

```bash
./server
```

默认监听：

```text
http://127.0.0.1:8080/
```

### 守护进程运行

```bash
./server --daemon
```

或者：

```bash
./server -d
```

## 设计记录里的几个真实踩坑

这个项目有意思的地方之一，是它保留了很多很真实的开发痕迹。比如：

- 命名空间没有写清楚，导致“明明是同一个类却被当成不同类型”
- 传值和传引用没想清楚，接口设计容易变乱
- `include` 管理不小心时，容易出现重复定义

这些问题在学习型项目里非常常见，也正因为踩过这些坑，后面的版本结构才越来越清楚。
